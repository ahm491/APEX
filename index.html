ما المقصود بـ Low-Code / No-Code؟What do Low-Code / No-Code mean?

التطوير منخفض الشيفرة نهجٌ لبناء البرمجيات يعتمد على واجهات مرئية (سحب وإفلات) وإعدادات تعريفية بدل كتابة آلاف الأسطر البرمجية. تكتب شيفرةً فقط عند الحاجة الفعلية.Low-code development is an approach to building software that relies on visual interfaces (drag and drop) and declarative settings instead of writing thousands of lines of code. You write code only when genuinely needed.

No-Code

بناء تطبيقات كاملة دون كتابة أي شيفرة — عبر الإعدادات والخيارات التعريفية فقط. مناسب لمعظم الحالات الشائعة.Building complete apps without writing any code — through settings and declarative options only. Fits most common cases.

Low-Code

معظم البناء مرئي وتعريفي، مع إمكانية إضافة شيفرة (SQL أو JavaScript) في النقاط التي تحتاج تخصيصاً دقيقاً.Most of the build is visual and declarative, with the option to add code (SQL or JavaScript) exactly where precise customization is needed.

Pro-Code

التطوير التقليدي بكتابة كل شيء يدوياً. مرونة قصوى، لكنه أبطأ وأعلى كلفة وصيانة. Low-code يغطي ٨٠٪ من الحاجة بجزء من الجهد.Traditional development, writing everything by hand. Maximum flexibility, but slower and costlier to build and maintain. Low-code covers 80% of needs with a fraction of the effort.

لماذا ظهر منخفض الشيفرة؟

الطلب على التطبيقات يفوق قدرة فرق التطوير التقليدية بكثير. المؤسسات تحتاج شاشات، لوحات، ونماذج داخلية بوتيرة متسارعة، بينما التطوير التقليدي بطيء ومكلف. جاء منخفض الشيفرة ليردم هذه الفجوة: يبني معظم التطبيق عبر واجهات مرئية وإعدادات تعريفية، فيصبح البناء أسرع وأقل عرضةً للأخطاء، ويُوسّع دائرة من يستطيعون بناء تطبيقات ذات جودة.

تعريف دقيق

تعريفي (Declarative) يعني أنك تصف «ماذا تريد» (مثلاً: تقرير على هذا الجدول، عمود التاريخ للقراءة فقط) بدل أن تكتب «كيف يُنفَّذ» سطراً بسطر. المنصّة تتكفّل بالتنفيذ.

  • الحوكمة وتضخّم التطبيقات: سهولة البناء تُنتج فوضى تطبيقات دون معايير ومراجعة وملكية واضحة.

ما هو Oracle APEX؟

اختصار لـ Oracle Application Express: منصّة أوراكل منخفضة الشيفرة الأشهر عالمياً لبناء تطبيقات مؤسسية آمنة وقابلة للتوسّع. ميزتها الجوهرية أنها تعمل داخل قاعدة بيانات أوراكل نفسها، وتبني تطبيقات ويب وجوّال من المتصفح مباشرةً دون بيئة تطوير محلية معقّدة.

مجاني مع قاعدة بيانات أوراكل يعمل من المتصفح ويب + جوّال + سطح مكتب مدعوم بالذكاء الاصطناعي (24.x) سحابة أو داخل المؤسسة

Why did low-code emerge?

Demand for applications far exceeds what traditional development teams can deliver. Organizations need screens, dashboards, and internal forms at an accelerating pace, while traditional development is slow and costly. Low-code emerged to close this gap: most of the application is built through visual interfaces and declarative settings, making delivery faster and less error-prone, and widening the circle of people who can build quality applications.

A precise definition

Declarative means you describe "what you want" (for example: a report on this table, with the date column read-only) instead of writing "how it runs" line by line. The platform takes care of the execution.

Read the numbers critically

The speed and code-size multiples above are vendor-reported figures (Oracle) under best-case conditions, and they vary widely by application type and team. Use them as a general direction, not a numeric promise to management — the real value shows up specifically in data-driven internal applications.

When it fits — and when it does not

✔ A strong fit

  • Data-driven internal applications (entry screens, reports, tracking dashboards).
  • Replacing shared Excel sheets with a secure web application.
  • Approval flows and enterprise workflow.
  • Portals and front-ends over an Oracle database or Fusion.

⚠ Think twice (the hard truths)

  • Highly unique UIs, graphics-heavy experiences, and latency-critical real-time systems.
  • The 20% wall: the last slice of highly customized requirements can cost more effort than the platform saved.
  • Governance and app sprawl: easy building produces application chaos without standards, review, and clear ownership.
  • Vendor lock-in: you build inside the Oracle ecosystem; moving away later is not free.
  • Licensing: APEX is "free" only with a licensed Oracle database; apex.oracle.com is for learning, within limits.

What is Oracle APEX?

Short for Oracle Application Express: Oracle's flagship low-code platform for building secure, scalable enterprise applications. Its defining trait is that it runs inside the Oracle database itself, building web and mobile applications directly from the browser with no complex local development environment.

Free with an Oracle database Runs from the browser Web + mobile + desktop AI-powered (24.x) Cloud or on-premises

Executive summary

APEX is not a hobbyist tool; it is an enterprise platform on which banks and governments run critical applications. It is well suited to building our internal application portfolio quickly and at minimal cost.

Hands-on — train your judgment on the fit decision

Before building anything, the most important decision is: does APEX fit this case? For each of the following, decide "fits" or "think twice", and justify:

  1. Project site visits tracking system (our project): visit entry, reports, and a status dashboard over a database.
  2. A real-time structural computation engine processing thousands of sensor readings per second with 3D rendering.
  3. A change-order approval board showing data from Oracle Fusion and managing an approval flow.

Hint: apply the rule — "data-driven + internal → fits" and "critical real-time or unique UI → think twice". (1 and 3 are strong fits; 2 is think twice.)

How does APEX work? (Architecture)

There is no heavy application server. The browser requests a page; the request passes through the ORDS layer (Oracle REST Data Services) to the APEX engine living inside the database, which generates the page from its stored definitions and returns ready HTML. This simplicity is why APEX is fast and inexpensive to run.

The main components of the environment

App Builder

The heart of the platform: build applications, their pages, and their components visually. You will spend most of your time here.

SQL Workshop

For managing data: browse tables, run SQL commands, generate schemas quickly, and load data.

Gallery & Team Dev

A gallery of ready starter and sample apps for learning, plus tools for organizing teamwork and tasks.

The Workspace and its roles

A workspace is a logical container grouping your developers, applications, and data schema. Inside it are three main roles:

Workspace Administratorمدير المساحة
Manages users, settings, and services at the workspace level.
Developerالمطوّر
Builds applications and edits pages and data — the role you will usually work in.
End Userالمستخدم النهائي
Uses published applications only, with no development privileges.

Hands-on — get a free workspace

  1. Open apex.oracle.com and choose Get Started for Free.
  2. Choose Request a free workspace and enter your email and a workspace name.
  3. A confirmation email arrives; activate the account and set a password.
  4. Sign in — you will find App Builder and SQL Workshop at the top. Explore both.

Caution: the free workspace is for learning only

The free apex.oracle.com workspace is meant for evaluation and learning — not for production or sensitive data. It may be deleted after a period of inactivity (60 days) and some settings are restricted. For production we use APEX on our own database or the APEX Service on OCI.

The basic building blocks of data

An application displays data stored in tables. Every table is rows and columns. Columns have types (text, number, date). Two kinds of keys tie tables together:

Primary Key (PK)المفتاح الأساسي
A column that uniquely identifies each row (like an employee number). Never repeats, never empty.
Foreign Key (FK)المفتاح الأجنبي
A column linking a row in one table to a row in another (employee → their department). The basis of relationships.
Constraintقيد
A rule the database enforces on the data (unique, not null, value range).

The SQL Workshop tools

Object Browser

A visual browser for tables, columns, and data — create and modify tables without writing SQL.

SQL Commands

An editor for running SQL queries and PL/SQL commands directly when needed.

Quick SQL

Write a shorthand, indentation-based schema and it generates the full table and relationship DDL for you.

Data Workshop

Load data from CSV, XLSX, XML, and JSON into new or existing tables.

Quick SQL — a full schema in minutes

Instead of hand-writing CREATE TABLE statements, write this shorthand (indentation means a master-detail relationship):

# Projects and their visits
projects
    name        vc80 /nn
    client      vc80
    site_visits
        visit_date date /nn
        engineer   vc80
        status     vc40
        findings   vc300

Notation: vc80 text of length 80, num number, date date, /nn mandatory (Not Null). Nesting under projects automatically creates the FK in site_visits.

Hands-on

  1. From SQL Workshop → Utilities → Quick SQL, paste the schema above.
  2. Review the preview, then click Generate SQL, then Save SQL Script and Run.
  3. Switch to Object Browser and confirm both tables exist.
  4. Alternatively: in Data Workshop, upload an Excel file you have; APEX creates a table from its columns automatically.

Practical tip

The fastest way to build a first application: start from an Excel file you already have via Data Workshop. Real data makes learning tangible and fast.

The three-pane Page Designer interface

Opening any page for editing shows three panes. Understanding each pane’s role makes your work fast and organized.

Left: the rendering tree

Rendering Tree — shows the page components hierarchically (regions → items → buttons). Click any element to select it.

Center: the layout pane

Right: the Property Editor

Property Editor — every property of the selected element. Here you set the title, source, condition, style… everything.

The seven page components

Every APEX page is built from these blocks. Memorize them well — they are the vocabulary of your new language.

Regionsالمناطق
Content containers on the page: a report, a form, a chart, static HTML… everything lives inside a region.
Itemsالحقول
Elements that enter/display values: text, select list, date, checkbox. Each holds a value readable in code (P1_NAME).
Buttonsالأزرار
Trigger actions: submit the page, navigate to another page, or fire a dynamic action.
Dynamic Actionsالإجراءات الديناميكية
Instant browser interactivity without a reload: show/hide an item, change a value, run on an event (change, click).
Processesالعمليات
Logic executed on the server when the page is submitted: save to the database, send email, call PL/SQL.
Validationsالتحقّقات
Check inputs before saving (required field, valid email). Block the save and show a message on failure.
Branchesالتفرّعات
Decide where the user goes after processing (usually back to the same page or to a results page).

A golden navigation rule

Anything happening instantly in the browser → a Dynamic Action. Anything needing the server/database (save, fetch) → a Process on submit. Internalize this distinction and everything becomes logical.

Page items and session state

This concept ties everything together: how APEX "remembers" values and carries them between pages. Every field on a page is an Item holding one value — like P7_PROJECT_ID holding the selected project in the change-order form. Its value is stored in Session State on the server, so it stays available after submit and when moving to another page.

Page itemP7_PROJECT_ID → Session statekept by the server → Query / logicreads the value

Why the bind variable matters

Writing :P7_PROJECT_ID in SQL passes the value bound, not spliced into the query text — which is exactly what prevents SQL injection automatically. This is "security built into the framework" made concrete: you get it by writing correctly, with no extra effort.

Interactive Report or Interactive Grid?

This is the key decision when presenting a data table. The simple rule: does the user only read, or edit?

Interactive Report — IR

For reading and analysis. Gives the user search, sort, filter, grouping, charts, export (Excel/PDF/CSV), and private saved reports. Does not edit data directly.

Interactive Grid — IG

For display and editing together. An Excel-like grid: edit cells in place, add/delete rows, drag columns, freeze panes. Ideal for multi-row data entry.

Modern presentation experiences

Faceted Search

A side panel of filters (as on shopping sites): the user picks a department, a status, a date range, and the results filter instantly. Excellent for large datasets.

Cards

Shows each record as a card with an image, title, and buttons instead of table rows. Ideal for project lists and site visits on mobile.

The page lifecycle: rendering, then processing

This is the most important concept in the stage. Every page has two phases: rendering when it opens, and processing when it is submitted. Each component runs in its own phase.

The order is sacred

Validations always run before processes. So never put save logic inside a validation, and never put input checks inside a process. Put everything in its right place and the system behaves reliably.

Lists of Values (LOV): stop the error at its root

Instead of letting the user type the project name by hand (and get it wrong), offer a dropdown of approved values. That is a List of Values.

Dynamic LOV

Sourced from a SQL query on a table (like the projects table). Updates automatically as the data changes. The most common kind.

Static LOV

Manually defined values (Draft/Approved/Rejected). For short lists that do not change.

  3. Add a validation: Amount is required and greater than zero. Write a clear message.
  4. Run, leave Amount empty, and click Save — watch processing stop and the message appear.

Charts with the Oracle JET engine

APEX uses the Oracle JET library for charts: interactive, responsive, and Arabic-friendly. Pick a type, bind it to a query, and it renders instantly.

Bar / column

For comparing categories (like visits per project). The clearest for management.

Line

For tracking a trend over time (monthly performance, revenue growth).

Pie / donut

For parts of a whole (budget distribution). Use with a small number of slices.

Anatomy of a Dynamic Action

A Dynamic Action follows a simple three-link logic: an event occurs, an optional condition is checked, then actions run for the true or false outcome.

More components worth trying

The Map for geographic locations, the Calendar for events and appointments, and the Timeline for event sequences. All are ready-made regions — bind a query and they appear instantly.

Hands-on — a dashboard and a Dynamic Action

  1. Create a new page and add a Chart → Bar region on "visits per project".
  2. Add two more regions to the same page to make a small dashboard.
  3. In the Stage-7 form: add a Dynamic Action on the Contract Type item, event Change.
  4. Make the True action Show the Salary item, and the False action Hide it. Run and try it.

Authentication vs Authorization

Two concepts many people confuse. The rule: authentication = who are you? and authorization = what may you do? The first is the entrance gate; the second is the room keys.

One practical example that settles the difference

Two employees sign in successfully (authentication passes for both). Yet the Approve Budget button appears for the manager only and is hidden from the employee (different authorization). Same door, different privileges.

Shared Components: set once, apply everywhere

Shared Components are settings at the level of the whole application, not a single page. Change them once and every page reflects it.

Navigation and menus

The side menu and top bar buttons — defined once, shown on every page automatically.

The Universal Theme

The visual identity: colors, templates, fonts. Change a color once and the whole app changes.

LOVs and data sources

Shared LOVs, data sources, settings — define once, reuse on any page.

Built-in security — protecting you with no effort

Session Managementإدارة الجلسات
APEX manages each user’s session securely and ends it automatically on inactivity.
SQL Injection protectionحقن SQL
Using bind variables blocks injection by default across APEX components.
XSS protectionCross-Site Scripting
APEX escapes output automatically to stop malicious scripts running in the browser.
Session State Protectionحماية حالة الجلسة
Prevents tampering with page values through the URL by signing sensitive values.

Security is built in — but do not neglect it

The built-in protections are strong, but they do not excuse you from setting authorization on every sensitive page and enabling session-state protection for important items. Security is a shared responsibility between you and the framework.

  2. Open the form page, select the Delete button, and set its Authorization property to the scheme.
  3. Sign in as a non-manager — watch the button disappear.
  4. Explore Shared Components → User Interface → Theme and try changing the color.

Integration over REST: connect your world

APEX does not live in isolation. Through REST Data Sources it consumes data from any other system and shows it in a report as if it were a local table — including connecting our data in Oracle Fusion.

Automation and workflows

Automations

Workflow

The deployment cycle: from development to production

You built a great app — how do you move it into real use safely without breaking what works? Through a controlled pipeline and working copies.

The golden deployment rule

Do not test on production, and never edit a live app directly. Export a copy, test it, then release it. While developing, use Debug mode to trace page execution line by line and find the source of any error, and APEX Advisor to check errors, performance, and security before every release.

Hands-on — export and a working copy

  1. From App Builder open your app, then Export/Import → Export Application and save the SQL file.
  2. Create a Working Copy of the app and try a large change in it without fear for the original.
  3. Run Utilities → Advisor and review its suggested improvements.
  4. If you have access: explore REST Data Sources and add a trial source.

Artificial intelligence in APEX 26.1

The highlight of the modern release. AI is no longer an add-on; it is woven into the development environment and into the applications you build.

The APEX AI Assistant

The APEX AI Assistant helps you while building: it suggests, explains, and speeds up creating components.

Natural-language data modeling

Describe what you want in words and it generates a data model or scripts — a huge acceleration for getting started.

Vector Search and RAG

AI Vector Search enables semantic search and apps that answer from your documents (RAG) — search by meaning, not just by keyword.

Newest in 26.1: AI-assisted development you can trust

Release 26.1 (May 2026) moved AI from mere "assistance" to governed, transparent generation — you own what is generated and can review and govern it.

APEXlang

A readable declarative language representing your app as text files (.apx) — exported, reviewed, and managed in version control (Git), and generated transparently by AI. The principle: "if you generated it, you own it."

AI Interactive Reports (NL2IR)

The user asks the report in natural language — "show me the overdue visits for project X" — and APEX applies the filtering and grouping automatically. It builds directly on the Interactive Reports we mastered in S6.

AI Agents

Agents that understand a request and act through explicitly authorized "tools" only (fetch data, run PL/SQL, a specific action) — smart automation inside safe limits you define, not open-ended privilege.

Directly relevant to our project

The findings column in SITE_VISITS fills with free-text notes. With vector search an engineer can ask "where did waterproofing problems recur?" and find results by meaning — even when the exact word never appears in the note.

  2. In App Builder try the APEX Assistant: ask it to create a report or explain an existing component.
  3. Conceptually: design a semantic search over the findings column that answers engineers about recurring note patterns across projects.

ابنِ نظام SLA لعملياتك في APEXBuild an SLA system for your processes in APEX

الهدف النهائي من التدريب: أن يصمّم الفريق بنفسه اتفاقيات مستوى الخدمة (SLA) لعمليات الشركة ويقيسها ويحسّنها. هذه الوحدة تبني نظام SLA كاملاً فوق نفس تطبيق الورشة — من الجدول إلى الإنذار المبكّر — بأربع مراحل تطبيقية.The training’s end goal: the team designs, measures, and improves Service Level Agreements (SLAs) for company processes by itself. ما SLA بلغة العمليات؟What is an SLA in process language?

اتفاقية مستوى الخدمة التزامٌ زمني قابل للقياس: «كل أمر تغيير عاجل يُستجاب له خلال ٤ ساعات ويُغلق خلال ٢٤». لبناؤها تحتاج ثلاثة أشياء فقط: طوابع زمنية على كل عملية (استلام/استجابة/إغلاق)، جدول مستهدفات حسب الأولوية، واستعلام مقارنة بينهما. كل ما بعد ذلك — لوحات وإنذارات وتحسين — يُبنى فوق هذه الثلاثية.A Service Level Agreement is a measurable time commitment: "every urgent change order gets a response within 4 hours and closes within 24". Building one needs only three things: timestamps on every process (received/responded/closed), a targets table by priority, and a comparison query between them. Everything after that — dashboards, alerts, improvement — is built on this triad.

٨

نموذج بيانات SLA: طوابع زمنية ومستهدفاتThe SLA data model: timestamps and targets

  1. في SQL Workshop › SQL Commands أضف الطوابع الزمنية والأولوية إلى أوامر التغيير، وأنشئ جدول المستهدفات:In SQL Workshop › SQL Commands, add the timestamps and priority to the change orders, and create the targets table:
-- ١) الطوابع الزمنية على العملية-- 1) Timestamps on the process
alter table change_orders add (
  received_at   timestamp default systimestamp,
  responded_at  timestamp,
  closed_at     timestamp,
  priority      varchar2(20) default 'NORMAL'
);

-- ٢) جدول مستهدفات SLA حسب الأولوية-- 2) SLA targets table by priority
create table sla_targets (
  id                number generated by default on null as identity primary key,
  priority          varchar2(20) not null unique,
  response_hours    number not null,
  resolution_hours  number not null
);
insert into sla_targets (priority, response_hours, resolution_hours) values ('URGENT', 4, 24);
insert into sla_targets (priority, response_hours, resolution_hours) values ('NORMAL', 24, 72);
commit;
  1. في نموذج أمر التغيير (المرحلة ٤) حوّل «الأولوية» إلى LOV ثابتة (URGENT/NORMAL)، واجعل عملية «تسجيل الاستجابة» تملأ responded_at بـsystimestamp عند أول ردّ.In the change-order form (phase 4), make Priority a static LOV (URGENT/NORMAL), and have a "record response" process fill responded_at with systimestamp on first reply.
  2. أدخل ٥–٦ أوامر تجريبية بأولويات وأزمنة مختلفة (بعضها متأخّر عمداً) لتختبر الحساب لاحقاً.Enter 5–6 test orders with mixed priorities and times (some deliberately late) to test the calculation later.
٩

استعلام الالتزام: من الطوابع إلى نسبةThe compliance query: from timestamps to a percentage

  1. هذا الاستعلام يحكم على كل أمر: داخل SLA أم خرق — جرّبه في SQL Commands:This query judges every order: within SLA or a breach — try it in SQL Commands:
select co.id, co.priority,
       round((cast(co.responded_at as date) - cast(co.received_at as date)) * 24, 1) as resp_hrs,
       t.response_hours as target_hrs,
       case when (cast(co.responded_at as date) - cast(co.received_at as date)) * 24
                 <= t.response_hours
            then 'IN_SLA' else 'BREACH' end as sla_status
from   change_orders co
join   sla_targets  t on t.priority = co.priority
where  co.responded_at is not null
  1. ونسبة الالتزام الإجمالية — الرقم الذي تطلبه الإدارة — سطر واحد فوق النتيجة نفسها:And the overall compliance rate — the number management asks for — is one line over the same logic:
select round(100 * avg(case when (cast(responded_at as date) - cast(received_at as date)) * 24
                                 <= t.response_hours then 1 else 0 end), 1) as compliance_pct
from   change_orders co
join   sla_targets t on t.priority = co.priority
where  co.responded_at is not null
  1. كرّر المنطق نفسه لمستهدف الإغلاق (closed_at مقابل resolution_hours).Repeat the same logic for the resolution target (closed_at vs resolution_hours).

  2. استعلامها: الأوامر المفتوحة التي اقترب موعد استجابتها — قبل الخرق لا بعده:Its query: open orders whose response deadline is near — before the breach, not after it:

دورة التحسين — SLA ليست لوحة تُعلَّقThe improvement cycle — an SLA is not a plaque on the wall
